# 위임 프롬프트 규칙

모델 선택보다 이 규칙이 결과를 더 크게 가른다. 실측에서 **같은 모델·같은 effort가 프롬프트 한 문단 차이로 완전히 정직해지거나 통째로 지어냈다.** 근거는 각 항목에 붙였고, 원 데이터는 `measurements.md`.

## 규칙 1 — 탈출구를 준다

부정적 결론을 낼 통로를 명시적으로 열어 준다. 리뷰라면 "결함이 없으면 정확히 `NO BUG FOUND`라고만 답하라", 지식 질의라면 "확인할 수 없으면 확인할 수 없다고 말하라"를 넣는다. 여기에 "없는 것을 지어내는 편이 침묵보다 나쁘다"를 덧붙인다.

**근거 (A/B).** 존재하지 않는 함수 `FilterEngine::applyDitherStage()`를 있는 것처럼 제시하고 노이즈 셰이핑 차수와 멤버 변수 이름을 캐물었다.

- 탈출구 없음 → luna low는 "3차"라고 단정하고 `ditherError`라는 멤버를 창작했으며 초기화 동작까지 서술했다. terra medium도 같은 방식으로 `m_noiseShapingErrors`를 만들어냈다.
- 탈출구 있음 → **같은 luna low가** 확인 불가 항목 넷을 열거하고 일반론만 답했다. 품질도 떨어지지 않았다. 오히려 같이 물어본 RBJ 증명에 ω₀≠0 단서가 붙어 더 정확해졌다.

리뷰 쪽에서도 같다. 계약을 조이고 `NO BUG FOUND` 탈출구를 준 깨끗한 코드에 luna low·luna medium·sol high·terra medium 네 조합 모두 정확히 그 한 줄만 냈다. 거짓 양성이 0이었다.

**주의.** 탈출구가 검출력을 죽이지 않는지 반드시 대조군으로 확인해야 한다. 위와 같은 계약에 결함을 심은 판을 넣었을 때 SOL과 Terra는 그대로 잡아냈다. 탈출구는 지어내기만 막고 진짜 결함은 놓치지 않았다.

## 규칙 2 — 산출물의 완결 조건을 못박는다

"이 파일 하나만으로 컴파일되어야 하니 필요한 헤더와 타입 별칭을 전부 포함하라"처럼 쓴다. 무엇을 포함하고 무엇을 빼야 하는지 애매하게 두면 절반이 어긋난다.

**근거 (A/B).** 구현 과제에서 "헤더를 다시 쓰지 말고 정의만 하라"고 썼더니 luna medium과 terra medium이 `using EqPoints = ...` 별칭을 통째로 빼 컴파일이 깨졌다(각각 29개, 26개 오류). 문자 그대로 따른 해석이다. 별칭 한 줄만 넣으니 둘 다 35/35를 통과했다. 로직은 처음부터 완벽했다.

같은 과제를 Workflow에서 "필요한 헤더와 타입 별칭을 전부 포함해 이 파일 하나로 컴파일되어야 한다"로 바꿔 위임했더니 별칭이 정상으로 들어왔고, 저널에서 꺼낸 코드가 그대로 컴파일돼 통과했다.

**교훈은 실력이 아니라 문장에 있다.** 위임 프롬프트의 모호함이 만든 실패의 비용은 전부 적용 단계의 Claude에게 온다.

## 규칙 3 — 범위 밖을 미리 잘라낸다

보고하지 않아도 되는 것을 계약으로 명시한다. 성립을 보장하는 불변식, 형식론적 UB, 스타일, 성능 같은 것들이다.

**근거.** 계약을 조이기 전에는 소견의 절반이 실무에서 도달할 수 없는 것들로 찼다. 32비트 오버플로우를 노린 `(frameCount, bufferLength, bufferOffset) = (2, 4294967295, 4294967294)` 같은 트리거가 그렇다(실제로는 버퍼 길이가 32Mi로 잘려 도달 불가). 좌결합 포인터 산술이 만드는 일시적 범위 밖 포인터도 마찬가지다. 계약에 "이건 보장된다, 보고하지 말라"를 넣자 소견이 진짜 결함에만 모였다.

덧붙여, 결함을 지목할 때 **무엇을 함께 제출해야 하는지**를 요구한다. 줄 번호, 문제가 되는 부분식 원문, 고친 부분식, 그것을 터뜨리는 구체적 입력, 그리고 무엇이 잘못되는지(출력 오류인지 크래시인지)를 요구하면 검증 가능한 소견만 돌아온다.

## 규칙 4 — 검증할 수 없는 것을 요구하지 않는다

릴레이 GPT는 저장소를 못 보고 아무것도 실행하지 못한다. 그러니 저장소 특정 사실(파일명, 함수명, 호출 순서, 빌드 플래그)은 **프롬프트 본문에 붙여** 준다. 붙이지 않은 것을 물으면 규칙 1의 상황이 된다.

실행 여부는 신고 줄로 확인한다.

```
VERIFIED: compiled=<yes|no> tested=<yes|no>
```

**근거.** 이 줄을 요구한 네 조합 모두 `compiled=no tested=no`로 정직하게 신고했다. 실행하지 않은 것을 했다고 주장한 사례가 없었다. 다만 이건 신고 줄을 명시적으로 요구했기 때문이다. 요구하지 않으면 확인할 방법이 없다.

**적용은 언제나 Claude가 실제로 한다.** GPT의 신고를 믿고 넘어가지 말고 컴파일과 테스트를 직접 돌린다. 그것이 게이트다.

## 요약 체크리스트

위임하기 전에 프롬프트를 훑는다.

- 부정적 결론을 낼 통로가 명시되어 있는가.
- 산출물이 무엇을 포함해야 완결인지 못박혀 있는가.
- 보고하지 않아도 되는 범위가 적혀 있는가.
- GPT가 볼 수 없는 것을 묻고 있지는 않은가. 물어야 한다면 본문에 붙였는가.
- 소견의 형식(줄 번호, 원문, 수정, 트리거)을 요구했는가.
- 실행 신고 줄을 요구했는가.
