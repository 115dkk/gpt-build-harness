# 오케스트레이션 루프

릴레이 형태의 정석 루프. Claude가 설계·적용·검증을 쥐고, GPT가 구현 노동을 낸다.

## 한 사이클
1. **Claude가 설계·범위를 확정**한다. 무엇을 만들지, 인터페이스(타입/시그니처/반환)가 어떤지 Claude가 못박는다. 프론트엔드는 특히 Claude가 확정(house rule).
2. **자기완결 프롬프트로 위임**한다. GPT는 대화 맥락·파일 접근이 없으므로, 기존 코드·스펙·기대 인터페이스·테스트 기준을 프롬프트 본문에 붙인다.
3. **반환물을 Claude가 적용**한다 — HTML 엔티티 복원, 파일로 심기, 모듈 등록(mod/handler), 설정 교정.
4. **빌드·테스트**한다. 실패하면 에러 로그를 그대로 GPT에 되먹여 재질의(핵심 루프). 성공까지 반복.
5. **커밋하고 교훈을 로그에 적는다.**

## 선(先)명세가 재질의를 줄인다 (이 토이의 핵심 발견)
두 사이클(config 파서, RBJ DSP) 모두 **재질의 0회**로 GREEN. 인터페이스를 Claude가 프롬프트에 못박은 덕이다(FilterLine 모델, command 시그니처, tolerance 테스트 기준까지). **하네스의 비용은 재질의가 아니라 선명세에 있다.** 잘 정의된 표준 알고리즘일수록 스펙만 정확하면 1샷으로 나온다. 재질의 루프는 스펙이 모호하거나 프로젝트 특수 규약(빌드 플래그, 사내 API)이 얽힐 때 나타난다 — 그럴수록 명세를 더 조여 위임하라.

## 적용 잔손질은 Claude 몫
릴레이는 코드를 텍스트로 주므로, 적용 단계의 잔손질이 전부 Claude에게 온다. 이 잔손질(엔티티 복원, 등록, identifier 교정, 빌드/테스트)이 곧 게이트다 — GPT 산출물이 저장소에 들어가기 전 Claude가 반드시 통과시킨다.

## 위임 단위를 사이클로 쪼갠다
큰 작업을 독립 사이클로 나눈다(파서 / DSP / …). 각 사이클은 인터페이스 확정 → 위임 → 적용 → 검증 → 커밋. **프론트(Claude)와 백엔드(GPT)를 병렬로** 굴리면 GPT가 도는 동안 Claude가 UI를 짜 대기 시간이 겹친다. 이 토이는 매 사이클 그렇게 했다.

## house rule
- **프론트엔드 디자인**: 설계는 Claude 확정, 구현·검토만 GPT 위임. GPT 감각이 나쁘다.
- **CI/CD**: 품질을 먼저 확인하기 전에는 위임 금지. GPT가 자주 깬다.
- 메인/워커 형태(게이트 없음)일 땐 사용자가 직접 감독.
