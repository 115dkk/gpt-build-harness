# 오케스트레이션 루프

릴레이 형태의 정석 루프. Claude가 설계·적용·검증을 쥐고, GPT가 구현 노동을 낸다.

## 한 사이클
1. **Claude가 설계·범위를 확정한다**. 무엇을 만들지, 인터페이스(타입/시그니처/반환)가 어떤지 Claude가 못박는다. 프론트엔드는 특히 Claude가 먼저 확정해야 한다.
2. **자기완결 프롬프트로 위임한다**. GPT는 대화 맥락·파일 접근이 없으므로, 기존 코드·스펙·기대 인터페이스·테스트 기준을 프롬프트 본문에 붙인다.
3. **반환물을 Claude가 적용한다** — HTML 엔티티 복원, 파일로 심기, 모듈 등록(mod/handler), 설정 교정.
4. **빌드·테스트한다**. 실패하면 에러 로그를 그대로 GPT에 되먹여 재질의한다(핵심 루프). 성공까지 반복한다.
5. **커밋하고 교훈을 로그에 적는다.**

## 선명세가 재질의를 줄인다 (이 토이의 핵심 발견)
두 사이클(config 파서, RBJ DSP) 모두 **재질의 0회**로 성공했다. 인터페이스를 Claude가 프롬프트에 못박은 덕이다(FilterLine 모델, command 시그니처, tolerance 테스트 기준까지). **하네스의 비용은 재질의가 아니라 선명세에 있다.** 잘 정의된 표준 알고리즘일수록 스펙만 정확하면 1회 만에 나온다. 재질의 루프는 스펙이 모호하거나 프로젝트 특수 규약(빌드 플래그, 사내 API)이 얽힐 때 나타난다 — 그럴수록 명세를 더 조여 위임해야 한다.

## 적용 잔손질은 Claude 몫
릴레이는 코드를 텍스트로 주므로, 적용 단계의 잔손질이 전부 Claude에게 온다. 이 잔손질(엔티티 복원, 등록, identifier 교정, 빌드/테스트)이 곧 게이트다 — GPT 산출물이 저장소에 들어가기 전 Claude가 반드시 통과시킨다.

## 위임 단위를 사이클로 쪼갠다
큰 작업을 독립 사이클로 나눈다(파서 / DSP / …). 각 사이클은 인터페이스 확정 → 위임 → 적용 → 검증 → 커밋으로 흐른다. **프론트(Claude)와 백엔드(GPT)를 병렬로** 굴리면 GPT가 도는 동안 Claude가 UI를 짜 대기 시간이 겹친다. 이 토이는 매 사이클 그렇게 했다.

## 디자인 위임 vs 렌더 번역 (프론트에서 GPT를 쓰는 법)
house rule은 프론트 **디자인 결정**을 GPT에 맡기지 말라는 것이지, 이미 확정된 디자인의 **구현 번역**까지 막는 게 아니다. 이 둘을 나누면 프론트가 무거운 작업에서도 GPT를 크게 쓸 수 있다.

- **디자인 위임(금지)**: 무엇을 어떻게 보이게 할지 GPT가 정하는 것. GPT 감각이 나쁘다.
- **렌더 번역(위임 가능)**: Claude(또는 기존 코드)가 확정한 시각을 다른 런타임으로 옮기는 것 — 확정된 Qt QPainter를 canvas 2D로, 디자인 시안을 컴포넌트로. 값·색·좌표를 1:1로 옮기는 노동이라 GPT가 잘한다.
- **실전(Rack 목업)**: Qt 페인터가 디자인의 단일 원천 → Claude가 레이아웃·색 토큰·CSS 프레임을 확정 → GPT가 페인터를 canvas로 번역(chrome/controls 2병렬) → Claude가 배선·검증한다. 프론트지만 house rule과 정합한다.
- **GUI 없는 검증**: 렌더 함수를 순수하게(그리기 컨텍스트만 받게) 유지하면 스텁/헤드리스 canvas(node-canvas)로 창 없이 결과를 PNG로 확인할 수 있다 — computer-use 불필요.

## house rule
- **프론트엔드 디자인**: 설계는 Claude가 확정하고, 구현·검토(와 위 렌더 번역)만 GPT에 위임한다. GPT 감각이 나쁘다.
- **CI/CD**: 품질을 먼저 확인하기 전에는 위임 금지. GPT가 자주 깬다.
- 메인/워커 형태(게이트 없음)일 때는 사용자가 직접 감독해야 한다.
